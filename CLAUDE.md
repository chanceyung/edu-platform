# CLAUDE.md — 智慧教学平台 开发规则

> 本文件是本项目（面向初中学校的 AI 教学减负系统，"老师的 AI 数字员工"）的**最高开发准则**，所有 AI 协作与人工开发都必须遵守。
> 核心精神：**扎实开发，严禁胡乱开发。** 每一行代码都要能追溯到需求、经过测试、符合规范。

---

## 〇、铁律（不可违背，违背即视为破坏项目）

1. **License 纯净**：严禁引入 AGPL / GPL / SSPL / ELv2 / BSL 或任何限制性许可的代码与依赖。每个新依赖引入前**必须查证 license**并记录。
2. **学之思只参考设计，严禁抄码**：学之思（xzs）是 AGPL，只允许参考其功能/数据模型/流程设计，**严禁复制、改写、翻译其任何代码**进入本项目。
3. **AI 必可追溯**：所有 AI/Agent 操作必须写入 `ai_task_log`/`ai_grading_log`（含输入/输出/模型/耗时/操作人），可审计、可回滚。无日志的 AI 功能禁止上线。
4. **测试先行**：核心业务逻辑必须先写测试再实现（TDD）。禁止无测试的业务代码进入主干。
5. **禁止半成品进主干**：`TODO`/`FIXME`/`注释掉的代码`/`console.log`/`print 调试`/未实现的功能，禁止合入主干。
6. **不擅自引入技术**：新增框架/中间件/大依赖须经技术评审并在本文档登记，不得随手 `import`。
7. **模块边界神圣**：模块间严禁跨层直接调用数据库或内部实现，必须走定义好的接口/API。
8. **改动遵循现有模式**：匹配周围代码风格、命名、分层，不顺手"改进"无关代码（外科手术式改动）。

## 一、项目使命与定位

- **做什么**：面向初中学校的 AI 教学减负系统，核心用户是**教师**（替老师干活，不是让老师填表）。
- **商业逻辑**：用"给老师减负"获客（老师爱用），用"学情/提分数据"卖学校（校长买单）。
- **产品范式**：**Agent-first**——老师下达目标，Agent 自主规划+执行+交付，老师核验。
- 完整规划见 `docs/产品规划.md`（多端 / 16 功能模块 / 数据模型 / 开发计划 / AI 深度设计 / Agent 架构）。

## 二、技术栈（强制约束，不得擅改）

| 层 | 选型 |
|---|---|
| 后端 | Spring Boot 3.x + JDK 17 + MyBatis-Plus + MySQL 8 + Redis |

## 三、项目结构与模块边界

```
edu-platform/
├── backend/                # 后端（Maven 多模块）
│   ├── gateway/            # API 网关 / 启动模块（聚合启动）
│   ├── ruoyi-base/         # 脚手架：用户/角色/菜单/部门/字典/日志/代码生成（直接用 RuoYi）
│   ├── school-org/         # 学校组织：校/年级/班/学期/师生/任课（自研）
│   ├── question-bank/      # 题库（参考学之思设计，自写）
│   ├── exam-engine/        # 考试引擎（参考 sg-exam 代码 + 学之思判分模型）
│   ├── homework/           # 作业（自研）
│   ├── ai-service/         # AI 能力 + Agent 编排 + 信任内核（自研，核心）
│   ├── analytics/          # 成绩学情分析（自研）
│   └── home-school/        # 家校沟通（自研）
├── frontend/
│   ├── teacher-workbench/  # PC 教师工作台（核心）
│   ├── admin/              # PC 管理后台
│   ├── student-pc/         # PC 学生端
│   ├── mobile-student/     # 移动学生端（uniapp）
│   └── mobile-parent/      # 移动家长端（uniapp）
├── ai-platform/            # Agent 平台：工具协议(MCP)/编排/任务管理/信任内核
├── deploy/                 # docker-compose / k8s / nginx / 配置
└── docs/                   # 产品规划、开发规范、数据字典、API 文档
```

**模块边界规则**：
- 每个模块有独立 `pom.xml` 和明确的业务职责。
- 模块依赖只能单向（上层依赖下层公共），**禁止循环依赖**。
- 跨模块调用走 `api` 子包定义的接口，不直接依赖他模块的 `service/impl`。
- 公共能力放 `ruoyi-base` 或独立的 `common`，不放业务模块。

## 五、Agent-first 开发规则

1. **系统功能先 MCP 化**：任何要被 Agent 调用的能力，先按 `ai-platform` 的工具协议封装为 Tool（含入参/出参 schema/权限/校验），再实现。
2. **Agent 必带信任内核**：每个 Agent 执行必须——动作分级（自动/需确认/禁止）、全程 `ai_task_log`、可中断、可回滚、关键节点人在回路。
3. **交付物写回系统**：Agent 成果须通过工具写入数据库/发消息/生成文件，不得只返回文本。
4. **混合架构**：关键路径用状态机/工作流兜底，不纯靠 LLM 自由发挥。
5. **失败降级**：Agent 失败须降级为人工/规则，不得静默吞错。

## 六、代码规范

- **命名**：后端类名 PascalCase、方法变量 camelCase、常量 UPPER_SNAKE；包名全小写。前端组件 PascalCase、变量 camelCase。
- **分层**：Controller（只做参数校验/转发）→ Service（业务）→ Mapper/Repository（持久化）。禁止 Controller 直连 DB，禁止业务逻辑写在 Controller。
- **注释**：公开接口/复杂逻辑/业务规则必须有注释；禁止"废话注释"（如 `// 获取用户` 配 `getUser()`）。
- **异常**：统一异常处理，禁止吞异常，业务异常用明确错误码。
- **日志**：关键操作打 INFO，异常打 ERROR 带堆栈；禁止用 `System.out`。
- **DTO/VO/Entity 分离**：禁止把数据库实体直接返回给前端。

---

## 七、数据库规范

- 表名前缀按模块：`org_*`（学校组织）/ `qbank_*`（题库）/ `exam_*`（考试）/ `hw_*`（作业）/ `ai_*`（AI）等。
- 所有表必须有：`id`(bigint 主键)、`tenant_id`(租户)、`create_time`/`update_time`/`creator`/`updater`/`is_deleted`（软删）。
- 字段命名 snake_case；枚举用 tinyint + 字典表，不用魔法数字。
- **变更必须走迁移脚本**（Flyway/Liquibase），禁止手改生产库；脚本不可逆操作须评审。
- 千分制分值（score×1000 存 int）遵循学之思设计；富文本统一外置 `text_content`。

---

## 八、API 规范

- RESTful，路径 `/api/{module}/{resource}`，版本前缀 `/api/v1`。
- 统一返回结构 `{code, message, data}`，统一错误码。
- 所有接口必须有参数校验 + 权限注解 + 入参/出参 DTO。
- 接口文档用 Knife4j/Swagger 自动生成，保持与代码同步。

---

## 九、测试要求

- **核心业务 TDD**：先写失败测试，再实现至通过。
- **覆盖率门槛**：核心模块（school-org/exam-engine/homework/ai-service）行覆盖 ≥ 80%，其他 ≥ 60%。
- 测试分层：单元测试（mock 依赖）+ 集成测试（真实 DB/Redis）+ 关键链路 E2E。
- AI 能力必须有**评测集**（批改准确率、组卷合理性等），回归不达标阻塞发布。

---

## 十、质量门禁与提交流程

- **分支**：`main`（生产）/ `develop`（集成）/ `feature-*` / `fix-*` / `release-*`。禁止直接推 `main`。
- **提交信息**：Conventional Commits（`feat:` / `fix:` / `refactor:` / `test:` / `docs:` / `chore:`），含模块前缀，如 `feat(exam-engine): 新增随机组卷`。
- **PR/合并**：CI 全绿（编译+测试+lint+覆盖率）+ 至少 1 人 Code Review，方可合并。
- **CI 必检**：编译、单元测试、lint（后端 checkstyle/spotless，前端 eslint+prettier）、license 扫描（拒绝 AGPL/GPL）、覆盖率门槛、TODO 扫描（拒绝带 TODO 进主干）。
- **发布**：打 tag，附 changelog，灰度上线。

## 十一、禁令清单（出现即打回）

- ❌ 使用领课教育(roncoo-education)作底座或参考代码——它是 **AGPL-3.0**（常被误判为"许可证友好"，实则 SaaS 触发开源义务），严禁引入
- ❌ 复制学之思或任何 AGPL/GPL 项目的代码
- ❌ AI/Agent 操作无日志、不可追溯
- ❌ 无测试的业务代码
- ❌ TODO/FIXME/注释代码/调试输出进主干
- ❌ 擅自引入新依赖/框架（未评审登记）
- ❌ 跨模块直接调用内部实现、循环依赖
- ❌ Controller 写业务逻辑、直连 DB
- ❌ 把数据库实体直接返回前端
- ❌ 手改数据库不走迁移脚本
- ❌ 硬编码魔法数字、密钥、配置
- ❌ 提交真实密钥/凭据到仓库

## 十二、AI 协作约定（给 Claude 及开发者）

- 改动前**先读**相关模块的现有代码与测试，理解模式。
- 遵循现有命名、分层、风格，**不顺手重构无关代码**。
- 任何改动都要有对应测试；优先让现有测试通过，不破坏既有行为。
- 涉及 AI/Agent，必须同时实现信任内核（日志/分级/可回滚），缺一不可。
- 不确定时**先问，不猜**；不臆造 API、字段、库的能力。
- 报告要诚实：测试失败就说失败，跳过的步骤要说明，不夸大成果。
